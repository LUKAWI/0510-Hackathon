## 1. 问题全景

赛题要求把 7 本医学教材压缩到不到 2.1 本的精华，同时教学效果不打折。这听起来像是一个简单的"删重复"操作，实际上涉及五个相互纠缠的子问题：

| 子问题 | 核心难点 | 与其他问题的耦合 |
|--------|---------|-----------------|
| 知识点粒度定义 | 切多细才算一个"知识点"？ | 直接影响重复判定和压缩比计算 |
| 重复判定标准 | 措辞不同但含义相同，算重复吗？ | 依赖粒度定义，决定压缩空间 |
| 教学连贯性保障 | 删完之后知识链条还完整吗？ | 约束压缩的上限 |
| 30% 压缩比实现 | 怎么从 100% 压到 30%？ | 前三个问题的综合结果 |
| RAG 分块策略 | 切块大小影响检索质量 | 与粒度定义相互独立但共享元数据 |

这些问题不是线性的，而是一个循环依赖：粒度定义影响重复判定，重复判定影响压缩空间，压缩又反过来要求粒度调整。我们的策略是先定粒度，再做对齐，最后在教学连贯性约束下执行压缩。

## 2. 知识点粒度定义

### 2.1 问题描述

教材内容是连续文本，但知识图谱需要离散节点。从一段 500 字的段落里，应该提取 1 个知识点还是 5 个？粒度太粗，图谱信息量不足，跨教材对齐也困难；粒度太细，图谱变成毛线球，节点数爆炸，整合成本指数增长。

### 2.2 难点分析

**难点一：没有客观标准。** "动作电位"是一个知识点，还是应该拆成"动作电位的产生机制""动作电位的传导""动作电位的特征"三个？不同教师会给出不同答案。

**难点二：粒度影响下游所有环节。** 如果把"免疫系统"当作一个节点，那跟《生理学》里的"免疫系统"直接匹配；但如果拆成"T 细胞""B 细胞""抗体"等子节点，匹配精度更高，但计算量也更大。

**难点三：教材本身粒度不统一。** 有的教材一节讲一个概念，有的一节讲五个。用统一规则切分，结果会偏向某种教材的写法。

### 2.3 解决方案：三级粒度体系

采用"章节 → 知识单元 → 知识原子"三级结构，由粗到细：

```
章节（Chapter）
  └── 知识单元（Knowledge Unit）  ← 图谱节点的主粒度
        └── 知识原子（Knowledge Atom）  ← 最小语义单元
```

**知识单元**是我们做图谱节点、跨教材对齐、整合决策的主要粒度。定义如下：

- 一个知识单元对应一个独立的、可被定义的概念、原理、方法或现象
- 通常对应教材中的一个小节或一个自然段落群
- 每个知识单元有且仅有一个核心主题
- 字数范围：200～800 字（太短缺乏上下文，太长需要拆分）

**知识原子**是知识单元的子结构，用于精细对齐和引用溯源：

- 一个定义、一条定理、一个公式、一个实验步骤
- 不单独作为图谱节点，但保留在知识单元的元数据中
- 用于 RAG 检索时的精确定位

```python
def validate_knowledge_unit(unit: dict) -> tuple[bool, str]:
    """校验知识单元的粒度是否合理"""
    content = unit["content"]

    # 太短：可能是片段，不是完整知识单元
    if len(content) < 100:
        return False, "内容过短，可能是片段而非独立知识单元"

    # 太长：可能包含多个知识点，需要拆分
    if len(content) > 1500:
        return False, "内容过长，可能包含多个知识点"

    # 缺少定义
    if len(unit["definition"]) < 10:
        return False, "定义过于简短"

    # 关键词太少：可能主题不明确
    if len(unit["keywords"]) < 1:
        return False, "缺少关键词，主题可能不明确"

    return True, "OK"
```

## 3. 重复判定标准

### 3.1 问题描述

"白细胞"和"白 blood 细胞"和"leukocyte"是同一个概念吗？《生理学》讲"炎症"侧重机制，《病理学》讲"炎症"侧重病理变化，它们是重复的还是互补的？

重复判定的错误会带来两种后果：
- **假阳性（误判为重复）**：丢失有价值的独特内容
- **假阴性（漏判重复）**：压缩比达不到 30% 的要求

### 3.2 难点分析

**难点一：语义相似 ≠ 内容重复。** 两本教材都讲"动作电位"，但一本侧重机制，一本侧重临床意义。从概念名看是重复的，从内容看是互补的。

**难点二：表述差异巨大。** 同一个概念在不同教材中的名称可能完全不同（如"程序性细胞死亡" vs "凋亡"），或者用不同的例子来解释同一原理。

**难点三：部分重叠。** 两个知识单元可能 70% 内容相同，30% 不同。合并还是分开？

### 3.3 解决方案：三层判定漏斗

采用"名称匹配 → 语义相似度 → LLM 深度判定"三层漏斗，逐层筛选：

```
全部知识单元对
    ↓ [第一层：名称/关键词匹配]
候选对（快速过滤明显不重复的）
    ↓ [第二层：Embedding 语义相似度]
疑似重复对（相似度 > 0.75）
    ↓ [第三层：LLM 深度判定]
确认重复对（含判定理由）
```

#### 第一层：名称/关键词快速匹配

```python
def name_keyword_match(unit_a: dict, unit_b: dict) -> float:
    """基于名称和关键词的快速匹配，返回 0-1 分数"""
    # 名称完全相同
    if unit_a["name"] == unit_b["name"]:
        return 1.0

    # 名称包含关系
    if unit_a["name"] in unit_b["name"] or unit_b["name"] in unit_a["name"]:
        return 0.8

    # 关键词重叠率
    kw_a = set(unit_a["keywords"])
    kw_b = set(unit_b["keywords"])
    if not kw_a or not kw_b:
        return 0.0
    overlap = len(kw_a & kw_b) / min(len(kw_a), len(kw_b))
    return overlap * 0.6  # 关键词匹配权重较低
```

#### 第二层：Embedding 语义相似度

```python
from sentence_transformers import SentenceTransformer
import numpy as np

class SemanticMatcher:
    def __init__(self, model_name: str = "BAAI/bge-small-zh-v1.5"):
        self.model = SentenceTransformer(model_name)

    def compute_similarity(self, unit_a: dict, unit_b: dict) -> float:
        """计算两个知识单元的语义相似度"""
        # 用 name + definition 作为语义表示
        text_a = f"{unit_a['name']}：{unit_a['definition']}"
        text_b = f"{unit_b['name']}：{unit_b['definition']}"

        embeddings = self.model.encode([text_a, text_b])
        similarity = np.dot(embeddings[0], embeddings[1]) / (
            np.linalg.norm(embeddings[0]) * np.linalg.norm(embeddings[1])
        )
        return float(similarity)

    def find_candidates(
        self,
        units: list[dict],
        threshold: float = 0.75
    ) -> list[tuple[str, str, float]]:
        """找出所有相似度超过阈值的知识单元对"""
        candidates = []
        for i in range(len(units)):
            for j in range(i + 1, len(units)):
                sim = self.compute_similarity(units[i], units[j])
                if sim >= threshold:
                    candidates.append((units[i]["id"], units[j]["id"], sim))
        return sorted(candidates, key=lambda x: -x[2])
```

### 3.4 阈值调优策略

阈值不是固定的，需要根据实际数据调整：

| 参数 | 初始值 | 调整方向 | 调整依据 |
|------|--------|---------|---------|
| Embedding 相似度阈值 | 0.75 | 低→高 | 假阳性太多则提高，压缩比不够则降低 |
| LLM 判定置信度阈值 | 0.8 | 低→高 | 误合并多则提高 |
| 跨学科阈值 | 0.85 | 比同学科高 0.1 | 不同学科的同名概念更可能是互补而非重复 |

## 4. 教学连贯性保障

### 4.1 问题描述

把 7 本教材压缩到 30%，必然会删除大量内容。但如果删除了某个知识点 A，而知识点 B 的定义里写着"参见 A"，或者学习 C 之前必须先掌握 A，那教学逻辑链就断了。

教学连贯性是压缩的硬约束：在满足 30% 压缩比的同时，不能让知识图谱出现"悬空引用"或"断裂的前置依赖"。

### 4.2 难点分析

**难点一：依赖关系是隐性的。** 教材不会在每个概念后面标注"本概念依赖以下前置知识"。依赖关系藏在行文中，比如"在了解了静息电位的基础上，我们可以理解动作电位……"

**难点二：跨章节依赖。** 第五章的概念可能依赖第二章的内容。章节内的压缩容易控制，跨章节的约束容易遗漏。

**难点三：不同教材的组织逻辑不同。** A 教材先讲结构再讲功能，B 教材反过来。整合时按谁的逻辑走？

### 4.3 解决方案：依赖图 + 连贯性检查器

#### 4.3.1 构建知识依赖图

在提取知识单元的同时，提取它们之间的依赖关系：

```python
DEPENDENCY_EXTRACTION_PROMPT = """

{units_list}

[
  {
    "source": "依赖方的知识单元 ID",
```

#### 4.3.2 连贯性检查器

在执行删除/合并决策后，运行连贯性检查：

```python
class CoherenceChecker:
    def __init__(self, dependency_graph: dict[str, list[str]]):
        """
        dependency_graph: {node_id: [prerequisite_ids]}
        """
        self.graph = dependency_graph

    def check_coherence(self, removed_ids: set[str]) -> list[dict]:
        """
        检查删除某些节点后，是否有节点的前置依赖被破坏。
        返回所有断裂的依赖关系。
        """
        violations = []
        for node_id, prereqs in self.graph.items():
            if node_id in removed_ids:
                continue  # 节点本身被删除，跳过

            broken = [p for p in prereqs if p in removed_ids]
            if broken:
                violations.append({
                    "node_id": node_id,
                    "broken_prerequisites": broken,
                    "severity": self._calc_severity(node_id, broken)
                })

        return sorted(violations, key=lambda x: -x["severity"])

    def _calc_severity(self, node_id: str, broken: list[str]) -> float:
        """
        严重程度计算：
        - 被多少其他节点依赖（影响范围）
        - 断裂的前置依赖占比
        """
        total_prereqs = len(self.graph.get(node_id, []))
        if total_prereqs == 0:
            return 0.0

        # 断裂比例
        break_ratio = len(broken) / total_prereqs

        # 影响范围：有多少节点依赖当前节点
        dependents = sum(
            1 for prereqs in self.graph.values()
            if node_id in prereqs
        )

        return break_ratio * (1 + dependents * 0.1)

    def suggest_fixes(self, violations: list[dict]) -> list[dict]:
        """
        对每个断裂的依赖，建议修复方案：
        1. 恢复被删除的前置节点
        2. 找到替代的前置节点
        3. 在保留节点中添加简要说明
        """
        fixes = []
        for v in violations:
            fixes.append({
                "node_id": v["node_id"],
                "options": [
                    {"action": "restore", "target": v["broken_prerequisites"]},
                    {"action": "add_brief", "description": "在节点内容中添加前置知识的简要说明"},
                    {"action": "find_alternative", "description": "查找可替代的前置知识点"}
                ]
            })
        return fixes
```

#### 4.3.3 压缩约束求解

将连贯性检查集成到压缩决策中，形成约束优化问题：

```python
def compress_with_coherence(
    units: list[dict],
    dependency_graph: dict[str, list[str]],
    target_ratio: float = 0.3,
    max_iterations: int = 10
) -> dict:
    """
    在满足教学连贯性约束的前提下，将内容压缩到目标比例。

    策略：
    1. 先删除明确的重复内容
    2. 检查连贯性，修复断裂
    3. 如果还没达到目标，合并互补内容
    4. 再次检查连贯性
    5. 如果仍不达标，删除低重要度的独立节点
    """
    total_chars = sum(len(u["content"]) for u in units)
    target_chars = total_chars * target_ratio
    current_chars = total_chars

    removed_ids: set[str] = set()
    merge_decisions: list[dict] = []

    # 第一轮：删除重复
    duplicate_pairs = find_duplicate_pairs(units)
    for pair in duplicate_pairs:
        to_remove = choose_which_to_remove(pair)
        removed_ids.add(to_remove)
        current_chars -= get_char_count(units, to_remove)

    # 检查连贯性
    checker = CoherenceChecker(dependency_graph)
    violations = checker.check_coherence(removed_ids)

    # 修复断裂
    for fix in checker.suggest_fixes(violations):
        best_option = select_best_fix(fix)
        if best_option["action"] == "restore":
            removed_ids.discard(best_option["target"])
            current_chars += get_char_count(units, best_option["target"])

    # 第二轮：合并互补内容
    if current_chars > target_chars:
        complementary_pairs = find_complementary_pairs(units, removed_ids)
        for pair in complementary_pairs:
            merged = merge_units(pair)
            merge_decisions.append(merged)
            current_chars -= pair.saved_chars

    return {
        "removed_ids": removed_ids,
        "merge_decisions": merge_decisions,
        "final_char_count": current_chars,
        "compression_ratio": current_chars / total_chars,
        "coherence_violations": checker.check_coherence(removed_ids)
    }
```

## 5. 30% 压缩比实现

### 5.1 问题描述

赛题明确要求："整合后保留的内容总字数不超过原始总字数的 30%"。7 本教材压缩到不到 2.1 本的量。

这不只是删删减减那么简单。30% 是一个很激进的目标，意味着要砍掉 70% 的内容，同时"教学效果不打折"。

### 5.2 难点分析

**难点一：压缩比的计算基准不明确。** "原始总字数"是指 7 本教材的全部字数，还是只算正文（去掉目录、索引、版权页）？"整合后字数"是指最终输出的全部内容，还是只算去重后的知识单元？

**难点二：30% 可能不够，也可能太多。** 如果 7 本教材有 60% 内容高度重复，那 30% 很容易达到；如果内容差异很大，30% 可能意味着删除有价值的独特内容。

**难点三：压缩比和教学完整性的冲突。** 为了凑 30% 而删除重要内容，违背了"教学效果不打折"的要求。

### 5.3 解决方案

#### 5.3.1 明确计算基准

```python
def calculate_compression_ratio(
    original_units: list[dict],
    final_units: list[dict],
    mode: str = "strict"
) -> dict:
    """
    计算压缩比。

    mode:
    - "strict": 原始字数 = 全部知识单元内容之和
    - "normalized": 原始字数 = 去除明显非教学内容后的字数
    """
    original_chars = sum(len(u["content"]) for u in original_units)

    if mode == "normalized":
        # 去除版权页、目录、索引等非教学内容
        # 这些已经在解析阶段标记过
        original_chars = sum(
            len(u["content"])
            for u in original_units
            if not u.get("is_frontmatter", False)
        )

    final_chars = sum(len(u["content"]) for u in final_units)

    return {
        "original_chars": original_chars,
        "final_chars": final_chars,
        "ratio": final_chars / original_chars if original_chars > 0 else 0,
        "reduction": 1 - (final_chars / original_chars) if original_chars > 0 else 0,
        "mode": mode
    }
```

#### 5.3.2 分层压缩策略

不是一刀切，而是按内容重要度分层处理：

```
第一层：去重（合并重复内容）
  → 预期压缩 30-40%（如果教材间重复率高）

第二层：精简（保留核心，删除展开说明）
  → 预期压缩 15-25%

第三层：重组（将分散的相关内容合并为紧凑单元）
  → 预期压缩 5-10%

第四层：裁剪（删除低重要度的独立内容）
  → 预期压缩 5-15%（最后手段）
```

#### 5.3.3 重要度评估

每个知识单元需要一个重要度分数，用于决定保留优先级：

```python
importANCE_SCORING_PROMPT = """
```

#### 5.3.4 压缩比动态调控

```python
def adaptive_compression(
    units: list[dict],
    duplicate_pairs: list[tuple[str, str]],
    target_ratio: float = 0.3,
    tolerance: float = 0.02
) -> dict:
    """
    自适应压缩：根据实际数据调整压缩策略。

    如果去重后已经接近目标，不再做激进裁剪。
    如果去重后还差很远，需要更激进的策略。
    """
    # 先做去重
    after_dedup = apply_deduplication(units, duplicate_pairs)
    dedup_ratio = len(after_dedup) / len(units)

    if dedup_ratio <= target_ratio + tolerance:
        # 去重后已经达标，不需要额外裁剪
        return {"strategy": "dedup_only", "units": after_dedup}

    if dedup_ratio <= target_ratio + 0.15:
        # 差距不大，精简展开说明即可
        return {"strategy": "dedup_and_simplify", "units": simplify(after_dedup)}

    # 差距较大，需要重组和裁剪
    scored = score_importance(after_dedup)
    return {"strategy": "full_compression", "units": compress_aggressive(scored, target_ratio)}
```

## 6. RAG 分块策略

### 6.1 问题描述

RAG 的第一步是把教材切成小块（chunk）。切块大小直接影响检索质量：太小，缺乏上下文，检索到的片段无法回答问题；太大，噪声多，检索精度下降。

赛题建议 500-800 字/块，50-100 字重叠。但这只是通用建议，针对医学教材可能需要调整。

### 6.2 难点分析

**难点一：医学知识的完整性。** 一个病理过程的描述可能跨越多个段落。如果在中间切断，检索到的片段可能只有"前半段"或"后半段"，无法回答完整问题。

**难点二：章节结构的影响。** 教材的章节结构本身就是一种知识组织。按固定字数切块会破坏这种结构。

**难点三：知识单元和 chunk 的关系。** 我们已经提取了知识单元，chunk 应该跟知识单元对齐吗？

### 6.3 解决方案：语义感知切块

不用固定字数切块，而是用知识单元作为 chunk 的基础：

```python
class SemanticChunker:
    """
    语义感知切块器：以知识单元为基础，结合章节结构进行切块。
    """
    def __init__(
        self,
        min_chunk_size: int = 300,
        max_chunk_size: int = 1000,
        overlap_size: int = 80
    ):
        self.min_size = min_chunk_size
        self.max_size = max_chunk_size
        self.overlap = overlap_size

    def chunk(self, chapter: dict, knowledge_units: list[dict]) -> list[dict]:
        """
        切块策略：
        1. 如果知识单元大小合适（min-max），直接作为一个 chunk
        2. 如果知识单元太小，合并相邻的小单元
        3. 如果知识单元太大，按段落拆分，保留重叠
        """
        chunks = []
        buffer = []
        buffer_size = 0

        for unit in knowledge_units:
            unit_size = len(unit["content"])

            if unit_size > self.max_size:
                # 先 flush buffer
                if buffer:
                    chunks.append(self._make_chunk(buffer, chapter))
                    buffer = []
                    buffer_size = 0

                # 大单元单独切分
                sub_chunks = self._split_large_unit(unit, chapter)
                chunks.extend(sub_chunks)

            elif buffer_size + unit_size > self.max_size:
                # buffer 满了，flush
                chunks.append(self._make_chunk(buffer, chapter))
                buffer = [unit]
                buffer_size = unit_size

            else:
                buffer.append(unit)
                buffer_size += unit_size

        if buffer:
            chunks.append(self._make_chunk(buffer, chapter))

        return chunks

    def _split_large_unit(self, unit: dict, chapter: dict) -> list[dict]:
        """按段落拆分大单元，保留重叠"""
        paragraphs = unit["content"].split("\n\n")
        chunks = []
        current = []
        current_size = 0

        for para in paragraphs:
            if current_size + len(para) > self.max_size and current:
                chunks.append({
                    "content": "\n\n".join(current),
                    "metadata": {
                        "chapter": chapter["title"],
                        "unit_id": unit["id"],
                        "unit_name": unit["name"],
                        "is_partial": True
                    }
                })
                # 保留重叠
                overlap_text = current[-1] if current else ""
                current = [overlap_text, para]
                current_size = len(overlap_text) + len(para)
            else:
                current.append(para)
                current_size += len(para)

        if current:
            chunks.append({
                "content": "\n\n".join(current),
                "metadata": {
                    "chapter": chapter["title"],
                    "unit_id": unit["id"],
                    "unit_name": unit["name"],
                    "is_partial": bool(len(chunks) > 0)
                }
            })

        return chunks
```

### 6.4 分块大小选择依据

| 分块大小 | 优点 | 缺点 | 适用场景 |
|---------|------|------|---------|
| 300 字 | 检索精度高 | 上下文不足 | 事实性问题（定义、分类） |
| 500 字 | 平衡精度和上下文 | 中庸 | 通用场景 |
| 800 字 | 上下文充足 | 噪声多 | 解释性问题（机制、原理） |
| 1200 字 | 完整语境 | 精度低 | 综合性问题（对比、分析） |

**我们的选择：** 默认 500 字，但允许 300-1000 的弹性范围。知识单元自然边界优先于固定字数。

## 7. 跨教材语义对齐

### 7.1 问题描述

不同教材对同一概念的表述可能完全不同。"细胞凋亡"在有的教材里叫"程序性细胞死亡"，有的叫"apoptosis"。需要一种机制来识别这些"不同名字的同一个东西"。

### 7.2 难点分析

**难点一：同义词和别名。** 医学术语有大量同义词、缩写、中英文混用。

**难点二：上下位概念混淆。** "免疫应答"和"特异性免疫应答"是包含关系，不是重复关系。

**难点三：跨学科同名概念。** "应激"在生理学和心理学中含义不同。

### 7.3 解决方案：多信号融合对齐

```python
class CrossBookAligner:
    """
    跨教材知识对齐器：融合多种信号判断两个知识点是否等价。
    """
    def __init__(self, embedding_model, llm_client):
        self.embedder = embedding_model
        self.llm = llm_client

    def align(self, units_a: list[dict], units_b: list[dict]) -> list[dict]:
        """
        对齐两本教材的知识单元。
        返回对齐结果列表。
        """
        alignments = []

        for ua in units_a:
            for ub in units_b:
                score = self._compute_alignment_score(ua, ub)
                if score > 0.6:  # 候选阈值
                    alignments.append({
                        "unit_a": ua["id"],
                        "unit_b": ub["id"],
                        "score": score,
                        "signals": self._get_signals(ua, ub)
                    })

        return alignments

    def _compute_alignment_score(self, ua: dict, ub: dict) -> float:
        """
        多信号融合评分：
        1. 名称相似度（权重 0.3）
        2. 定义语义相似度（权重 0.4）
        3. 关键词重叠（权重 0.15）
        4. 上下文相似度（权重 0.15）
        """
        name_sim = self._name_similarity(ua["name"], ub["name"])
        def_sim = self._embedding_similarity(ua["definition"], ub["definition"])
        kw_sim = self._keyword_overlap(ua["keywords"], ub["keywords"])
        ctx_sim = self._embedding_similarity(ua["content"][:200], ub["content"][:200])

        return (
            name_sim * 0.3 +
            def_sim * 0.4 +
            kw_sim * 0.15 +
            ctx_sim * 0.15
        )

    def _name_similarity(self, name_a: str, name_b: str) -> float:
        """名称相似度：精确匹配 > 包含关系 > 编辑距离"""
        if name_a == name_b:
            return 1.0
        if name_a in name_b or name_b in name_a:
            return 0.8
        # 编辑距离归一化
        max_len = max(len(name_a), len(name_b))
        if max_len == 0:
            return 0.0
        edit_dist = self._levenshtein(name_a, name_b)
        return 1.0 - (edit_dist / max_len)

    def _embedding_similarity(self, text_a: str, text_b: str) -> float:
        """语义相似度"""
        emb = self.embedder.encode([text_a, text_b])
        return float(np.dot(emb[0], emb[1]) / (
            np.linalg.norm(emb[0]) * np.linalg.norm(emb[1])
        ))

    def _keyword_overlap(self, kw_a: list[str], kw_b: list[str]) -> float:
        """关键词重叠率"""
        set_a, set_b = set(kw_a), set(kw_b)
        if not set_a or not set_b:
            return 0.0
        return len(set_a & set_b) / min(len(set_a), len(set_b))
```

## 8. LLM 输出可靠性

### 8.1 问题描述

LLM 是这个系统的核心引擎，但它的输出不可靠：可能输出格式错误的 JSON、可能编造不存在的知识点、可能给出不一致的判定结果。

赛题明确禁止"编造教材中不存在的知识点或引用"和"伪造或篡改教材出处"。

### 8.2 难点分析

**难点一：JSON 输出格式不稳定。** LLM 有时会在 JSON 里加注释、用单引号、遗漏逗号。

**难点二：幻觉问题。** LLM 可能"补充"教材中没有的内容，特别是当上下文不完整时。

**难点三：一致性问题。** 同一对知识单元，调用两次 LLM，可能得到不同的判定结果。

### 8.3 解决方案

#### 8.3.1 输出解析与容错

```python
import json
import re

def parse_llm_json(raw_output: str) -> dict | list | None:
    """
    容错解析 LLM 输出的 JSON。
    处理常见的格式问题。
    """
    # 尝试直接解析
    try:
        return json.loads(raw_output)
    except json.JSONDecodeError:
        pass

    # 尝试提取 JSON 块（LLM 有时会加 markdown 代码块标记）
    json_match = re.search(r'```(?:json)?\s*([\s\S]*?)```', raw_output)
    if json_match:
        try:
            return json.loads(json_match.group(1))
        except json.JSONDecodeError:
            pass

    # 尝试修复常见问题
    cleaned = raw_output.strip()
    # 去掉尾部逗号
    cleaned = re.sub(r',\s*}', '}', cleaned)
    cleaned = re.sub(r',\s*]', ']', cleaned)
    # 单引号替换为双引号
    cleaned = cleaned.replace("'", '"')

    try:
        return json.loads(cleaned)
    except json.JSONDecodeError:
        return None
```

#### 8.3.3 输出校验

```python
def validate_extraction_result(result: dict, source_text: str) -> tuple[bool, list[str]]:
    """
    校验 LLM 提取结果是否可信。
    返回 (是否通过, 问题列表)
    """
    issues = []

    # 检查提取的内容是否真的在原文中出现过
    if result.get("content"):
        # 简单的子串检查（生产环境可以用更模糊的匹配）
        content_words = set(result["content"].replace("，", " ").replace("。", " ").split())
        source_words = set(source_text.replace("，", " ").replace("。", " ").split())
        overlap = len(content_words & source_words) / max(len(content_words), 1)
        if overlap < 0.3:
            issues.append("提取内容与原文重叠度过低，可能存在幻觉")

    # 检查定义是否合理
    if result.get("definition") and len(result["definition"]) > 500:
        issues.append("定义过长，可能是内容而非定义")

    # 检查分类是否在预设范围内
    valid_categories = {"概念", "原理", "方法", "现象", "结构", "过程"}
    if result.get("category") and result["category"] not in valid_categories:
        issues.append(f"分类 '{result['category']}' 不在预设范围内")

    return len(issues) == 0, issues
```

## 9. 大文件处理与内存管理

### 9.1 问题描述

7 本教材，每本几百页，总大小约 826MB。把它们全部加载到内存是不现实的。特别是 PDF 解析，PyMuPDF 之类的库默认会把整页加载到内存。

### 9.2 难点分析

**难点一：PDF 解析的内存占用。** 一本 500 页的 PDF，每页解析后可能产生 10KB 的文本，全书就是 5MB。加上图片、表格等，内存占用更大。

**难点二：Embedding 批量生成。** 通义千问 API 有并发限制，一次发太多请求会被限流。需要控制批量大小。

**难点三：知识图谱的内存占用。** 如果 7 本教材每本提取 200 个知识单元，总共 1400 个节点。加上依赖关系，图结构本身不大，但加上元数据和向量，内存占用会增长。

### 9.3 解决方案

#### 9.3.1 流式解析

```python
async def parse_pdf_streaming(file_path: str, callback) -> None:
    """
    流式解析 PDF：逐页解析，不把整本书加载到内存。
    每解析完一页，调用 callback 处理结果。
    """
    import pymupdf  # PyMuPDF

    doc = pymupdf.open(file_path)
    total_pages = len(doc)

    for page_num in range(total_pages):
        page = doc[page_num]
        text = page.get_text("text")

        # 过滤页眉页脚
        text = filter_header_footer(text, page_num, total_pages)

        if text.strip():
            await callback({
                "page": page_num + 1,
                "content": text,
                "char_count": len(text)
            })

        # 释放页面资源
        page = None

    doc.close()
```

## 10. 知识图谱增量更新

### 10.1 问题描述

用户可能先上传 3 本教材，构建图谱，过一会再上传 4 本。这时候需要增量更新图谱，而不是从头重建。

教师通过对话修改整合决策后，图谱也需要实时更新。

### 10.2 难点分析

**难点一：增量对齐的复杂度。** 新教材的知识单元需要跟已有的所有知识单元做对齐，计算量随教材数增长。

**难点二：图谱一致性。** 修改一个节点可能影响多个关系。需要保证图谱在任何时刻都是一致的。

**难点三：版本管理。** 教师修改决策后，需要能回退到之前的版本。

### 10.3 解决方案

#### 10.3.1 增量对齐流程

```python
async def incremental_align(
    new_units: list[dict],
    existing_units: list[dict],
    existing_alignments: list[dict]
) -> list[dict]:
    """
    增量对齐：只对新知识单元与已有单元做对齐。
    """
    new_alignments = []

    for new_unit in new_units:
        best_match = None
        best_score = 0.0

        for existing in existing_units:
            score = compute_alignment_score(new_unit, existing)
            if score > best_score:
                best_score = score
                best_match = existing

        if best_score > 0.8:
            # 高度相似，合并
            new_alignments.append({
                "action": "merge",
                "new_unit": new_unit["id"],
                "existing_unit": best_match["id"],
                "score": best_score
            })
        elif best_score > 0.6:
            # 中等相似，标记为待人工确认
            new_alignments.append({
                "action": "review",
                "new_unit": new_unit["id"],
                "existing_unit": best_match["id"],
                "score": best_score
            })
        else:
            # 无匹配，作为新节点加入
            new_alignments.append({
                "action": "add",
                "new_unit": new_unit["id"]
            })

    return new_alignments
```

#### 10.3.2 决策版本管理

```python
class DecisionVersionManager:
    """
    管理整合决策的版本历史，支持回退。
    """
    def __init__(self):
        self.versions: list[dict] = []
        self.current_version: int = 0

    def save_snapshot(self, decisions: list[dict], description: str) -> int:
        """保存当前决策状态的快照"""
        version_id = len(self.versions)
        self.versions.append({
            "id": version_id,
            "timestamp": datetime.now().isoformat(),
            "description": description,
            "decisions": decisions.copy()
        })
        self.current_version = version_id
        return version_id

    def rollback(self, version_id: int) -> list[dict]:
        """回退到指定版本"""
        if version_id < 0 or version_id >= len(self.versions):
            raise ValueError(f"版本 {version_id} 不存在")

        self.current_version = version_id
        return self.versions[version_id]["decisions"].copy()

    def get_diff(self, version_a: int, version_b: int) -> dict:
        """获取两个版本之间的差异"""
        dec_a = {d["decision_id"]: d for d in self.versions[version_a]["decisions"]}
        dec_b = {d["decision_id"]: d for d in self.versions[version_b]["decisions"]}

        added = [d for did, d in dec_b.items() if did not in dec_a]
        removed = [d for did, d in dec_a.items() if did not in dec_b]
        modified = [
            {"old": dec_a[did], "new": dec_b[did]}
            for did in dec_a if did in dec_b and dec_a[did] != dec_b[did]
        ]

        return {"added": added, "removed": removed, "modified": modified}
```

## 11. 多轮对话上下文管理

### 11.1 问题描述

教师可以跟系统进行多轮对话，追问、修改决策、要求解释。系统需要维护对话上下文，理解指代关系（"这个知识点""刚才那个"），并能根据反馈修改整合结果。

### 11.2 难点分析

**难点一：指代消解。** "为什么把它们合并了？"中的"它们"指什么？

**难点二：上下文窗口限制。** 通义千问的上下文窗口有限，对话太长需要截断或摘要。

**难点三：对话与图谱的联动。** 教师说"把 A 和 B 分开"，系统需要找到对应的决策并修改图谱。

### 11.3 解决方案

#### 11.3.1 对话状态管理

```python
class ConversationState:
    """
    管理多轮对话的状态。
    """
    def __init__(self, max_history: int = 20):
        self.history: list[dict] = []
        self.max_history = max_history
        self.context_entities: dict[str, str] = {}  # 实体指代映射
        self.pending_actions: list[dict] = []  # 待执行的操作

    def add_message(self, role: str, content: str, metadata: dict | None = None):
        """添加消息到历史"""
        self.history.append({
            "role": role,
            "content": content,
            "timestamp": datetime.now().isoformat(),
            "metadata": metadata or {}
        })

        # 超过上限时，对旧消息做摘要
        if len(self.history) > self.max_history:
            self._summarize_old_messages()

    def get_context_window(self, max_tokens: int = 3000) -> list[dict]:
        """
        获取适合发送给 LLM 的上下文窗口。
        优先保留最近的消息，对旧消息做摘要。
        """
        recent = self.history[-10:]  # 最近 10 条
        older = self.history[:-10]

        if older:
            summary = self._summarize(older)
            return [{"role": "system", "content": f"之前的对话摘要：{summary}"}] + recent

        return recent

    def resolve_reference(self, text: str) -> str:
        """
        解析指代词，替换为具体实体。
        "为什么把它们合并了" -> "为什么把'动作电位'和'膜电位变化'合并了"
        """
        # 简单的规则匹配（生产环境可以用 LLM 做指代消解）
        replacements = {
            "它们": self._find_recent_entities(2),
            "这个": self._find_recent_entities(1),
            "那个": self._find_recent_entities(1),
        }

        for ref, entity in replacements.items():
            if ref in text and entity:
                text = text.replace(ref, f"'{entity}'", 1)

        return text
```

## 12. 性能瓶颈与优化

### 12.1 问题描述

赛题有明确的性能要求：RAG 问答首 token < 3s，向量检索 < 200ms，知识图谱渲染 500 节点 < 1s。这些指标在教材数量增加后可能达不到。

### 12.2 瓶颈分析

| 环节 | 潜在瓶颈 | 影响 |
|------|---------|------|
| PDF 解析 | 大文件 IO + 文本提取 | 上传后等待时间长 |
| LLM 调用 | API 延迟 + 并发限制 | 知识提取和对齐耗时长 |
| Embedding 生成 | API 调用次数 | 索引构建慢 |
| 向量检索 | 数据库查询 | 问答响应慢 |
| 图谱渲染 | 前端渲染性能 | 大图卡顿 |

### 12.3 解决方案

#### 12.3.1 异步处理管道

```python
import asyncio
from enum import Enum

class ProcessingStage(Enum):
    UPLOADING = "uploading"
    PARSING = "parsing"
    EXTRACTING = "extracting"
    BUILDING_GRAPH = "building_graph"
    ALIGNING = "aligning"
    INDEXING = "indexing"
    DONE = "done"

class AsyncProcessor:
    """
    异步处理管道：将耗时操作放到后台执行。
    """
    def __init__(self):
        self.tasks: dict[str, dict] = {}

    async def process_textbook(self, file_path: str) -> str:
        """
        异步处理一本教材，返回任务 ID。
        前端通过轮询或 WebSocket 获取进度。
        """
        task_id = generate_task_id()
        self.tasks[task_id] = {
            "stage": ProcessingStage.UPLOADING,
            "progress": 0,
            "error": None
        }

        # 启动后台任务
        asyncio.create_task(self._process_pipeline(task_id, file_path))

        return task_id

    async def _process_pipeline(self, task_id: str, file_path: str):
        """处理管道的各个阶段"""
        try:
            # 阶段 1：解析
            self._update_stage(task_id, ProcessingStage.PARSING, 10)
            chapters = await parse_textbook(file_path)

            # 阶段 2：知识提取
            self._update_stage(task_id, ProcessingStage.EXTRACTING, 30)
            units = await extract_knowledge_units(chapters)

            # 阶段 3：构建图谱
            self._update_stage(task_id, ProcessingStage.BUILDING_GRAPH, 60)
            graph = await build_knowledge_graph(units)

            # 阶段 4：索引
            self._update_stage(task_id, ProcessingStage.INDEXING, 80)
            await build_vector_index(units)

            self._update_stage(task_id, ProcessingStage.DONE, 100)

        except Exception as e:
            self.tasks[task_id]["error"] = str(e)
```

```sql
    content,
    metadata,
    1 - (embedding <=> $1::vector) AS similarity
```
