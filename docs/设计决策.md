## 1. 用户登录功能

### 1.1 决策结论

**黑客松阶段暂不实现用户登录功能。**

### 1.2 决策理由

| 因素 | 分析 |
|------|------|
| 赛题要求 | 赛题未要求用户认证，核心关注点是教材知识整合能力 |
| 开发周期 | 黑客松时间有限，登录功能涉及前后端联调、会话管理、权限控制，投入产出比低 |
| MVP 优先 | 先跑通核心链路（文档上传、知识提取、RAG 问答、图谱展示），再迭代非核心功能 |
| 演示场景 | 评委关注的是知识整合效果，不是用户体系完整性 |

### 1.3 当前行为

系统以"单用户模式"运行：

- 所有上传的教材归入同一个全局知识库
- 对话历史存储在浏览器本地（localStorage 或内存），不持久化到后端
- 知识图谱对所有访问者可见，无权限隔离

### 1.5 未来升级方案

#### 第一阶段：基础认证

**目标**：支持用户注册、登录、会话保持。

**技术选型**：

| 组件 | 方案 | 理由 |
|------|------|------|
| 前端状态管理 | Zustand + persist 中间件 | 轻量、TypeScript 友好、支持持久化 |
| 后端认证 | FastAPI + JWT | 无状态 token，适合前后端分离架构 |
| 密码存储 | bcrypt 哈希 | 业界标准，安全性高 |
| Token 存储 | HttpOnly Cookie | 防 XSS 攻击 |

**实现步骤**：

1. 后端新增 `/api/auth` 路由模块
   - `POST /api/auth/register`：注册
   - `POST /api/auth/login`：登录，返回 JWT
   - `POST /api/auth/refresh`：刷新 token
   - `GET /api/auth/me`：获取当前用户信息

2. 数据库新增 `users` 表

   ```sql
   CREATE TABLE users (
     id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
     email VARCHAR(255) UNIQUE NOT NULL,
     password_hash VARCHAR(255) NOT NULL,
     display_name VARCHAR(100),
     created_at TIMESTAMP DEFAULT NOW(),
     updated_at TIMESTAMP DEFAULT NOW()
   );
   ```

3. 前端创建 `src/components/auth/` 目录，实现登录/注册表单
4. 前端创建 `src/stores/authStore.ts`，管理认证状态
5. 添加路由守卫 `AuthGuard`，保护需要登录的页面

#### 第二阶段：数据隔离

**目标**：每个用户的数据相互隔离。

**改造点**：

1. 教材表新增 `user_id` 外键
2. 对话历史表新增 `user_id` 外键
3. 后端 API 中间件注入当前用户 ID，所有查询自动过滤
4. 前端请求头携带 token，后端统一校验

```python
# FastAPI 依赖注入示例
from fastapi import Depends, HTTPException
from fastapi.security import HTTPBearer

security = HTTPBearer()

async def get_current_user(token = Depends(security)):
    payload = decode_jwt(token.credentials)
    if not payload:
        raise HTTPException(status_code=401, detail="Invalid token")
    return payload["user_id"]

@app.get("/api/documents")
async def list_documents(user_id: str = Depends(get_current_user)):
    return await db.fetch_documents(user_id=user_id)
```

#### 第三阶段：协作功能（远期）

**目标**：支持多人协作编辑知识图谱、共享教材库。

- 教材库共享：用户可将自己的教材库设为公开/私有
- 图谱协作：多人同时编辑同一知识图谱，冲突通过 OT 算法解决
- 评论系统：对知识节点添加讨论和批注

### 1.6 不变性约束

无论未来是否添加登录功能，以下设计原则保持不变：

1. **API Key 仅在后端使用**：通义千问等第三方服务的密钥不暴露给前端
2. **数据库连接走环境变量**：连接串不硬编码
3. **前端不直接访问数据库**：所有数据操作通过后端 API
4. **组件设计保持无状态**：UI 组件不依赖全局认证状态，通过 props 接收数据
